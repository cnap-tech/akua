# Example 03 — multi-env app

The shape most production workspaces actually use. A reusable **Package** is published to OCI; **Apps** reference it with per-environment typed inputs; **Environments** carry policy tier + budget + approvers. All authored in KCL — these are cluster-facing KRMs, so KCL is canonical and YAML is a derived view via `akua export`.

## Layout

```
03-multi-env-app/
├── akua.mod                       workspace root (declares members + shared deps)
├── akua.sum                       digest + signature ledger (machine-maintained)
├── package/                       reusable Package (would be OCI-published)
│   └── package.k
├── apps/
│   └── checkout/
│       └── app.k                  App KRM with typed per-env inputs + targets
├── environments/
│   ├── dev.k                      Environment KRM (tier/dev, no budget)
│   ├── staging.k                  tier/startup, $200/mo budget
│   └── production.k               tier/production, $1500/mo, required approvals
└── deploy/                        (generated by `akua render`)
    ├── dev/
    ├── staging/
    └── production/
```

## How the pieces connect

**Package** (`package/package.k`) — typed definition with `Input` schema. Published to OCI in production; inlined here for the example. Same shape as example 02.

**App** (`apps/checkout/app.k`) — references the Package by OCI ref. Carries one `spec.inputs.<env>` per environment, each a typed `pkg.Input`. No `$file:` sidecars; no YAML/KCL mix. All values typed; all edits land as typed-code PRs.

**Environment** (`environments/{dev,staging,production}.k`) — per-env scope. Each picks a policy tier, optionally a budget, optionally approvers and soak rules.

## Render

```sh
# Resolve deps → writes akua.sum
akua add

# Render every environment
akua render --all-envs --out=./deploy

# Or a single environment
akua render --env=production --out=./deploy/production
```

Each render pulls the Package's inputs for that environment, applies the environment's policy tier, and produces committable raw YAML under `deploy/<env>/`.

## Deriving YAML views

Reconcilers consume YAML. The `.k` files are authoritative; the YAML view is derived:

```sh
# Per-resource view (for kubectl diff / argo diff / flux diff)
akua export apps/checkout/app.k                 --format=yaml > apps/checkout/app.yaml
akua export environments/production.k           --format=yaml > environments/production.yaml
```

Check these YAML files in to git or don't; they regenerate deterministically. The **rule** is: never hand-edit YAML for cluster-facing KRMs — edit the `.k` and re-export.

## Flow for a change

1. Agent or human edits `apps/checkout/app.k` (e.g. bump production `replicas` from 5 to 7).
2. CI runs `akua check && akua lint && akua test && akua render --env=production`.
3. `akua policy check --tier=tier/production` — returns `allow` / `deny` / `needs-approval`.
4. If `needs-approval`: the review surface notifies approvers; human approves in browser.
5. PR merges; deploy repo gets updated YAML; Argo/Flux reconciles.
6. `App.status` reflects the reconciled state across all three environments; visible via `akua app get checkout`.
